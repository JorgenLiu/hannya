# 项目讨论总结

## 产品定位

Hennya 的目标是做一个可在 Windows 和 macOS 双端使用的桌面应用，帮助用户基于多份输入文档生成结构化、可交付的 PPT。

核心链路是：

1. 导入多份文档。
2. 通过多轮交互澄清汇报对象、目标、风格和页数。
3. 由 AI 先生成整份 deck 的规划，再生成每页内容。
4. 匹配内置模板或用户模板。
5. 通过稳定的渲染引擎生成 PPTX。

## 可行性判断

### 高可行部分

- 多文档导入与统一解析。
- 多轮对话式需求澄清。
- 先生成大纲，再生成每页内容。
- 基于内置模板稳定导出 PPTX。

### 真正的难点

- 多文档结论的真实性与来源追踪。
- 自动排版质量控制。
- 用户自带模板的结构理解和稳定适配。

结论是：产品方向可行，但第一阶段必须优先收敛边界，不要一开始承诺“任意模板完美适配”。

## 技术路线结论

### 第一阶段主技术栈

- Electron
- React + TypeScript + Vite
- Zustand
- SQLite
- PptxGenJS
- JSZip + fast-xml-parser
- DeepSeek API

### 为什么第一版优先 Electron

- Node 生态对 PDF、DOCX、PPTX、ZIP/XML 处理更成熟。
- Windows/macOS 打包和桌面集成链路更稳。
- 更适合快速验证复杂文档工作流。

## 核心架构判断

项目不应让模型直接输出 PPT 文件，而应让模型输出结构化 JSON，再由程序执行渲染。

推荐的核心中间层为：

- `DocumentUnit`: 文档标准化切片。
- `DeckPlan`: 演示文稿级规划。
- `SlideSpec`: 页面级内容结构。
- `TemplateSchema`: 模板结构抽象。

这意味着系统设计上应分层：

- 文档摄取与预处理
- 内容规划与生成
- 模板理解与匹配
- 渲染与导出
- 审校与回退

## 模板策略结论

### 第一优先级

建设高质量内置模板库，并标记：

- 适用场景
- 风格标签
- 支持的页面类型
- 内容密度

### 第二阶段

支持用户上传规整模板，解析后让 AI 做结构总结，再让用户确认模板用途和适配方式。

### 第三阶段

支持样例页标注学习，提高用户模板复用稳定性。

## Electron 性能结论

Electron 本身不是主要风险，风险在于重任务放错位置。

需要从一开始坚持的原则：

- 主进程只负责窗口、IPC、任务编排。
- CPU 密集任务放到 worker 或子进程。
- 大文本和大中间结果不常驻前端状态。
- 解析、模板分析、图表生成采用队列和缓存。
- 导出阶段只做装配，不做大规模即时计算。

## Python 的角色结论

当前不建议把 Python 作为第一版主运行时，但应该从架构上预留后续接入能力。

Python 更适合承担：

- PDF 清洗和结构恢复
- OCR
- 去重去噪
- 抽取式摘要
- 表格和版式分析

推荐方式是把 Python 作为后续可选 sidecar，而不是一开始就做成整个项目的硬依赖。

## 当前已经落地的内容

项目目录已经搭建完成，包含：

- Electron + React + TypeScript + `electron-vite` 桌面骨架
- 可直接启动的桌面 demo 壳
- preload 到 renderer 的 IPC 调用链路
- 共享类型契约与 bootstrap mock 数据
- AI、文档解析、模板引擎、渲染引擎接口定义
- 可行性、架构、性能、模板路线等文档

当前 demo 主要验证的是工程链路已经打通，而不是业务闭环已经完成。也就是说，桌面壳、共享契约和 UI 展示已经可运行，但真实文档导入、模型生成、模板解析和 PPTX 导出仍未实现。

## 当前推荐的下一步

1. 实现文档导入与 `DocumentUnit` 生成。
2. 接入 DeepSeek，先跑通 `DeckPlan` 和 `SlideSpec` 输出。
3. 先做 8 到 12 套内置模板的 schema 和匹配逻辑。
4. 再进入 PPTX 渲染与导出。
