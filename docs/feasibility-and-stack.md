# 可行性与技术栈建议

## 项目目标

构建一个可在 Windows 和 macOS 双端运行的桌面应用，帮助用户导入多份资料，通过多轮交互补齐意图，自动生成结构化 PPT，并支持内置模板和用户模板。

## 可行性结论

### 高可行部分

- 多文档导入与统一解析。
- 多轮对话澄清汇报对象、目标、页数、风格和重点。
- 先生成大纲，再生成每页内容，再渲染 PPT。
- 基于内置模板稳定导出可用 PPTX。

### 中高复杂度部分

- 从多文档中抽取可靠结论并维持来源引用。
- 将不同类型的内容映射到合适的页面版式。
- 控制文本长度、图表空间、表格尺寸，避免生成结果拥挤或失衡。

### 高风险部分

- 任意用户上传模板后，系统全自动理解其版式语义并稳定适配。
- 对复杂母版、动画、形状层级很多的模板进行高质量复用。
- 在不做来源约束的情况下生成“看起来合理”的结论，容易出现幻觉与事实漂移。

## 产品分层建议

### 第一层：MVP

- 支持 PDF、DOCX、Markdown、TXT 导入。
- 多轮问答补齐生成意图。
- 生成 8 到 20 页的标准商务汇报。
- 提供 8 到 12 套内置模板。
- 导出 PPTX。
- 每页附带来源信息或来源映射。

### 第二层：增强版

- 允许用户导入规整的 PPTX 模板。
- 对模板进行结构分析，并让用户二次确认用途和版式类型。
- 支持局部重写、指定页重生成、页面顺序调整。

### 第三层：高级版

- 模板智能学习。
- 企业模板库和品牌规范。
- 更强的图表自动生成与数据页设计。
- 团队协作、权限和审计。

## 推荐技术栈

### 桌面层

- `Electron`: 第一版首选，跨平台成熟，Node 生态对文档处理和打包更友好。
- `electron-builder` 或 `electron-forge`: 用于打包与分发。

### 前端层

- `React + TypeScript + Vite`
- `Zustand`: 轻量状态管理。
- `TanStack Query`: 如果后续有云端接口、任务轮询、草稿状态同步，会比较合适。

### 数据与存储

- `SQLite`: 存项目、任务记录、文档索引、模板元数据。
- `keytar`: 存 DeepSeek API Key 或后续其他凭据。
- 本地文件系统: 存用户文档缓存、生成结果、临时渲染产物。

### 文档解析

- `pdfjs-dist` 或 `pdf-parse`: PDF 文本提取。
- `mammoth`: DOCX 文本提取。
- `markdown-it` 或 `remark`: Markdown 解析。
- `xlsx`: 表格读取。

### PPT 生成与模板处理

- `PptxGenJS`: 生成 PPTX 的核心库。
- `JSZip + fast-xml-parser`: 解压和读取 PPTX 的 OOXML 结构，用于模板分析。
- 图表建议先转成 `SVG` 或 `PNG` 后再插入 PPT，降低兼容性风险。

### AI 接入

- 使用兼容 OpenAI 风格的 SDK 对接 DeepSeek。
- 强制模型输出结构化 JSON，并对输出做 schema 校验。

## 为什么第一版更建议 Electron

- 文档处理、ZIP/XML、PPTX 相关 Node 库成熟。
- Windows/macOS 打包链路稳定。
- 更适合快速验证复杂桌面工作流。
- 后续如要加入 Python sidecar、OCR 或本地服务，Electron 集成更直接。

## 不建议第一版做的事

- 承诺支持任意复杂模板的完美自动适配。
- 让模型直接输出可写入 PPTX 的底层结构。
- 做过多视觉动画和复杂协作能力。
- 同时引入 Electron、Python 重处理链路和大量后台服务，导致系统过早复杂化。
