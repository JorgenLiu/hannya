# Electron 性能与 Python 预处理策略

## Electron 的性能问题要不要现在考虑

要，而且应该从第一版架构就控制好，但不需要因为性能焦虑而放弃 Electron。

对这个项目来说，真正会拖慢体验的通常不是“Electron 本身”，而是以下几类问题：

- 在渲染进程里直接做大文档解析。
- 主进程承担过多 CPU 密集任务。
- 一次性把大量解析结果和预览数据塞进前端状态。
- 模板分析、图表生成、PDF 解析全部同步执行。
- 没有任务队列、缓存和增量处理机制。

## Electron 常见性能风险

### 风险 1：主进程阻塞

如果主进程同时做文档解析、ZIP 解压、模板 XML 分析，会导致窗口卡顿、菜单无响应、导出体验差。

解决方案：

- 主进程只负责窗口、IPC、文件对话框、任务调度。
- CPU 密集工作放到 `worker_threads` 或独立子进程。
- 长任务一律异步化，并通过事件回传进度。

### 风险 2：渲染进程内存膨胀

用户可能一次导入多个 PDF/DOCX，如果把原始文本、切片、预览图、图表配置都常驻前端状态，内存会快速上涨。

解决方案：

- 渲染层只拿当前任务必要信息。
- 大对象落盘到 SQLite 或本地缓存目录。
- 列表页、预览页采用分页和惰性加载。

### 风险 3：IPC 传输过重

如果一次通过 IPC 传输几万行文本、整份模板解析树或高分辨率图片，会出现明显延迟。

解决方案：

- IPC 只传任务 ID、摘要、增量结果。
- 大型中间产物写到本地缓存，前端按需读取。
- 对重复任务做 hash 缓存，避免重复解析。

### 风险 4：导出阶段耗时过长

PPT 导出时如果还要即时算版式、即时做图表、即时压缩图片，导出时间会很不稳定。

解决方案：

- 导出前预先完成内容裁剪和版式决策。
- 图表资源提前渲染成 SVG/PNG。
- 导出阶段只做最后的装配和写文件。

## 推荐的性能方案

### 架构方案

- `main process`: 负责窗口、任务编排、权限与系统集成。
- `renderer process`: 负责 UI、交互和轻量数据展示。
- `worker pool`: 负责文档解析、chunk 切分、模板分析、图表生成。
- `local storage`: 负责缓存原始文件、解析结果、预处理结果和导出产物。

### 工程实践

- 给每个导入文件生成内容 hash。
- 解析结果缓存到本地，避免重复消耗 CPU 和 token。
- 文档处理采用任务队列，限制并发数。
- 大文本只展示摘要，全文按需读取。
- 预览优先返回低成本草稿，再做后台优化。

### UI 体验策略

- 长任务必须显示明确的阶段进度。
- 允许取消任务和重试。
- 支持先出大纲，再逐页细化，而不是一次生成整份 PPT。
- 每页单独重生成，避免一次失败重跑全局。

## 如果后续要引入 Python，应该怎么放

你的判断是合理的。即使不做本地推理，Python 仍然适合在后期承担“降低 token 消耗、提高输入质量”的预处理工作。

但建议不要一开始把 Python 变成主依赖，而是按 sidecar 或可选处理服务接入。

## Python 适合承担的任务

### 1. 文档清洗与去噪

- 清理页眉页脚、重复页码、版权声明、目录噪声。
- 合并断行、恢复段落。
- 识别重复片段，避免重复送模型。

### 2. 文档结构恢复

- 更好地识别章节层级。
- 将 PDF 文本恢复成段落、标题、列表和表格。
- 对扫描件接入 OCR 流程。

### 3. Token 优化

- 先做抽取式摘要，而不是把全文直接发给模型。
- 先按主题聚类，再只把相关 chunk 送入生成链路。
- 做术语归一和数字表述标准化，减少上下文冗余。

### 4. 高价值预处理

- 表格抽取和结构化。
- 图片 OCR 与图注提取。
- 版式特征分析。

## Python 接入方式建议

### 方案 A：Node 主体 + Python sidecar

适合第二阶段接入。

- Electron/Node 仍是主应用。
- Python 作为本地子进程或本地 HTTP 服务运行。
- 只在需要时调用特定预处理能力。

优点：

- 保持主应用简单。
- 不强迫所有功能都依赖 Python。
- 便于逐步引入 OCR、PDF 结构恢复等能力。

缺点：

- 打包变复杂。
- Windows/macOS 双端环境管理要提前设计。

### 方案 B：Node 主体 + 后端预处理服务

适合你后期如果准备做团队版或 SaaS。

- 客户端上传文档。
- 服务端做预处理与索引。
- 客户端只拿清洗后的结构化结果。

优点：

- 客户端更轻。
- 统一预处理环境。
- 更利于持续优化和监控。

缺点：

- 需要服务端基础设施。
- 涉及用户文档上传的隐私与合规问题。

## 对当前项目的建议

### 第一阶段

- 纯 Electron/Node 落地 MVP。
- 把预处理接口抽象出来，但先用 TypeScript 实现基础版本。
- 从架构上预留 worker 和 sidecar 接口。

### 第二阶段

- 针对 PDF 清洗、OCR、结构恢复引入 Python 能力。
- 只对高成本文档类型启用 Python。
- 对预处理结果做缓存与版本化。

### 第三阶段

- 根据真实用户数据决定是否把预处理迁移到云端。

## 一个关键原则

减少 token 使用，靠的不是单纯“压缩文本”，而是先把输入变成更适合生成的结构化上下文。

也就是说，真正值得建设的是：

- 内容去重
- 段落切分
- 主题聚类
- 证据映射
- 抽取式摘要

这些能力无论将来由 TypeScript 还是 Python 实现，都会持续提升生成质量和成本效率。
