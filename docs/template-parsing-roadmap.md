# 模板解析与半自动适配路线

## 目标

支持两类模板能力：

- 内置模板的稳定匹配与渲染。
- 用户上传模板后的结构提取、用途判断和半自动适配。

## 原则

不要把“上传任意模板后完美自动适配”当成第一阶段承诺。

更现实的目标是：

1. 稳定提取模板结构。
2. 让系统给出用途判断。
3. 让用户进行一次确认。
4. 再生成可复用的 `TemplateSchema`。

## 技术实现思路

### 1. 读取 PPTX 结构

PPTX 本质上是一个压缩包。可以通过 `JSZip` 解压，然后读取：

- `ppt/presentation.xml`
- `ppt/slideMasters/*`
- `ppt/slideLayouts/*`
- `ppt/theme/*`

重点提取：

- 母版和布局数量
- 占位符类型
- 文本框、图片框、图表框边界
- 主题字体和颜色

### 2. 归一化布局对象

把每个布局转换成统一 `TemplateLayout`，包括：

- 适合的页面类型候选
- 标题区位置
- 正文区数量与面积
- 图片区和图表区位置
- 内容密度判断

### 3. AI 归纳模板用途

在解析出结构后，再把提取结果送给模型做归纳：

- 这是正式风格还是营销风格
- 更适合数据页、章节页还是图文页
- 是否适合高密度内容

注意模型读取的是结构摘要，而不是直接读二进制模板。

### 4. 用户确认环节

建议在产品上加一层确认界面，让用户确认：

- 主要适用场景
- 常用页面类型
- 是否允许高密度文本页
- 是否有必须保留的品牌元素

这一步能显著提高后续渲染质量。

## 推荐分阶段实施

### Phase 1

- 只支持内置模板。
- 把模板 schema 模型和匹配器跑通。

### Phase 2

- 支持用户上传规整模板。
- 解析主题、布局、占位符。
- 生成模板摘要并让用户确认。

### Phase 3

- 支持样例页标注学习。
- 根据用户给出的“封面页 / 图表页 / 总结页”训练出模板映射规则。

## 工程风险点

### 风险 1：占位符语义不足

有些模板只是普通文本框和形状堆出来的，PPT 自身不会告诉你“这个框是摘要区”。

对策：

- 结合位置、大小、层级做规则推断。
- 再让 AI 根据结构摘要做弱分类。

### 风险 2：复杂母版和嵌套样式

某些模板有大量装饰形状、分组对象和视觉元素，解析结果会很嘈杂。

对策：

- 优先聚焦可承载内容的对象。
- 背景装饰保留为视觉标签，而不是纳入内容框逻辑。

### 风险 3：渲染结果不稳

即使模板解析正确，内容过长也会让页面崩掉。

对策：

- `deck-engine` 必须具备字数预算和溢出回退机制。
- 模板 schema 中加入密度标记和容量估算。

## 当前建议

第一版路线应是：

- 先做高质量内置模板库。
- 先做少量典型 layout 类型。
- 先把 `TemplateSchema` 和布局匹配机制做稳。
- 再逐步接入用户模板的半自动适配。

这样做能保证你最早拿到的是“能稳定交付的生成效果”，而不是一个功能很多但结果不稳定的系统。
